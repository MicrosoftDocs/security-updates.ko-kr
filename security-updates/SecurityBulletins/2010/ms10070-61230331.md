Microsoft 보안 공지 MS10-070 - 중요
===================================

ASP.NET의 취약점으로 인한 정보 유출 문제점(2418042)
---------------------------------------------------

게시된 날짜: 2010년 9월 28일 화요일 | 업데이트된 날짜: 2011년 10월 27일 목요일

**버전:** 4.2

### 일반 정보

#### 요약

이 보안 업데이트는 공개적으로 보고되었으며 정보 유출을 허용할 수 있는 ASP .NET의 취약점을 해결합니다. 이 취약점 악용에 성공한 공격자는 View State와 같이 서버에서 암호화된 데이터를 읽을 수 있습니다. 이 취약점 악용에 성공한 경우 서버에서 암호화된 데이터를 해독 및 변조하는 데 사용할 수 있는 데이터 변조에도 이 취약점을 사용할 수 있습니다. Microsoft .NET Framework 3.5 서비스 팩 1 이전의 Microsoft .NET Framework 버전은 이 취약점의 파일 콘텐츠 노출 부분의 영향을 받지 않습니다.

이 보안 업데이트의 심각도는 Microsoft .NET Framework 1.0 서비스 팩 3을 제외한 지원되는 모든 ASP.NET 에디션에 대해 중요입니다. 자세한 내용은 이 섹션에서 영향을 받는 소프트웨어 및 영향을 받지 않는 소프트웨어를 참조하십시오.

이 보안 업데이트는 ASP.NET으로 암호화된 모든 데이터를 추가로 서명하여 취약점을 해결합니다. 취약점에 대한 자세한 내용은 다음 절, 취약점 정보에서 각 취약점의 자주 제기되는 질문 사항(FAQ)을 참조하십시오.

이 보안 업데이트는 [Microsoft 보안 권고 2416728](http://technet.microsoft.com/security/advisory/2416728)에서 처음 설명한 취약점도 해결합니다.

권장 사항. 이 업데이트를 가능한 빨리 적용하는 것이 좋습니다.

이 공지 뒷부분에 있는 검색, 탐지 도구 및 지침 섹션도 참조하십시오.

문제점. 이 보안 업데이트를 설치할 때 고객이 겪을 수 있는 현재까지 알려진 문제점은 [Microsoft 기술 자료 문서 2418042](http://support.microsoft.com/kb/2418042)에 나와 있습니다. 이 문서는 이러한 문제점에 대한 권장 해결 방법도 소개합니다.

#### 영향을 받는 소프트웨어 및 영향을 받지 않는 소프트웨어

다음 소프트웨어는 테스트를 거쳐 영향을 받는 버전 또는 에디션이 확인되었습니다. 다른 버전은 지원 기간이 끝났거나 영향을 받지 않습니다. 사용 중인 소프트웨어 버전에 대한 지원 기간을 확인하려면 [Microsoft 지원 기간 정책 웹 사이트](http://go.microsoft.com/fwlink/?linkid=21742)를 참조하십시오.

이 보안 업데이트와 관련된 자주 제기되는 질문 사항
-------------------------------------------------


이 보안 공지가 2011년 2월 22일에 개정된 이유는 무엇입니까? 
Microsoft는 Windows 7(32비트 시스템용) 서비스 팩 1, Windows 7(x64 기반 시스템용) 서비스 팩 1, Windows Server 2008 R2(x64 기반 시스템용) 서비스 팩 1 및 Windows Server 2008 R2(Itanium 기반 시스템용) 서비스 팩 1을 실행하는 시스템에 Microsoft .NET Framework 4.0(KB2416472) 업데이트 패키지를 제공하는 검색 변경 사항을 알리기 위해 이 보안 공지를 개정하였습니다. 이 검색 변경 사항은 Windows 7(32비트 시스템용) 서비스 팩 1, Windows 7(x64 기반 시스템용) 서비스 팩 1, Windows Server 2008 R2(x64 기반 시스템용) 서비스 팩 1 또는 Windows Server 2008 R2(Itanium 기반 시스템용) 서비스 팩 1을 설치한 후 Microsoft .NET Framework 4.0을 설치한 고객에게만 적용됩니다. 이미 시스템을 성공적으로 업데이트한 고객은 추가 조치를 취할 필요가 없습니다.

이 보안 공지가 2010년 12월 14일에 개정된 이유는 무엇입니까? 
Microsoft는 Microsoft .NET Framework 4.0에 대한 새 업데이트 패키지(KB2416472)가 제공됨을 알리기 위해 이 보안 공지를 개정했습니다. 이러한 새 패키지는 다른 업데이트의 설공적인 설치를 방해할 수 있는 설정 문제점을 수정합니다. 이 문제점의 영향을 받는 다른 제품이나 업데이트를 설치한 고객은 [Microsoft 기술 자료 문서 2473228](http://support.microsoft.com/kb/2473228)에서 자세한 내용을 참조하십시오. 이미 시스템을 성공적으로 업데이트한 고객은 추가 조치를 취할 필요가 없습니다.

이 공지의 영향을 받는 소프트웨어에 나열되지 않은 버전인 .NET Framework 3.0 서비스 팩 2를 설치했습니다. 이 경우 업데이트를 설치해야 합니까? 
이 공지는 .NET Framework 2.0 및 .NET Framework 3.5 기능 계층의 취약점을 설명합니다. .NET Framework 3.0 서비스 팩 2 설치 프로그램은 .NET Framework 2.0 서비스 팩 2 설치에서 연계되므로 .NET Framework 3.0 서비스 팩 2을 설치하면 .NET Framework 2.0 서비스 팩 2도 설치됩니다. 따라서 .NET Framework 3.0 서비스 팩 2를 설치한 고객은 .NET Framework 2.0 서비스 팩 2용 보안 업데이트를 설치해야 합니다.

.NET Framework 3.5를 설치했습니다. 추가 업데이트를 설치해야 합니까? 
이 공지는 .NET Framework 2.0 및 .NET Framework 3.5 기능 계층의 취약점을 설명합니다. .NET Framework 3.5 설치 프로그램은 .NET Framework 2.0 서비스 팩 1 설치와 .NET Framework 3.0 서비스 팩 1 설치 모두에서 연계됩니다. 따라서 .NET Framework 3.5를 설치한 고객은 .NET Framework 3.5용 업데이트 외에도 .NET Framework 2.0 서비스 팩 1용 보안 업데이트를 설치해야 합니다.

시스템에 .NET Framework의 다른 버전이 설치되어 있는지 확인하려면, 이 섹션의 뒷부분에서 FAQ 항목, "설치되어 있는 Microsoft .NET Framework의 버전은 어떻게 알 수 있습니까?"를 참조하십시오.

.NET Framework 3.5 서비스 팩 1을 설치했습니다. 추가 업데이트를 설치해야 합니까? 
이 공지는 .NET Framework 2.0 및 .NET Framework 3.5 기능 계층의 취약점을 설명합니다. .NET Framework 3.5 서비스 팩 1 설치 프로그램은 .NET Framework 2.0 서비스 팩 2 설치와 .NET Framework 3.0 서비스 팩 2 설치 모두에서 연계됩니다. 따라서 .NET Framework 3.5 서비스 팩 1을 설치한 고객은 .NET Framework 2.0 서비스 팩 2용 보안 업데이트도 설치해야 합니다.

시스템에 .NET Framework의 다른 버전이 설치되어 있는지 확인하려면, 이 섹션의 뒷부분에서 FAQ 항목, "설치되어 있는 Microsoft .NET Framework의 버전은 어떻게 알 수 있습니까?"를 참조하십시오.

설치되어 있는 Microsoft .NET Framework의 버전은 어떻게 알 수 있습니까? 
시스템에 여러 버전의 .NET Framework를 동시에 설치하고 실행할 수 있으며 순서에 관계 없이 버전을 설치할 수 있습니다. 현재 설치되어 있는 .NET Framework의 버전을 확인하는 방법은 여러가지입니다. 자세한 내용은 [Microsoft 기술 자료 문서 318785](http://support.microsoft.com/kb/318785)를 참조하십시오.

시스템에 설치된 Microsoft .NET Framework 버전에 대해 두 가지 업데이트가 나열되어 있습니다. 두 업데이트를 모두 설치해야 합니까? 
예. 고객은 시스템에 설치된 소프트웨어에 대해 제공되는 모든 업데이트를 적용해야 합니다.

이러한 보안 업데이트를 특정 순서대로 설치해야 합니까? 
아니요. 하나의 .NET Framework 버전에 대해 여러 업데이트가 제공될 경우 원하는 순서대로 설치할 수 있습니다. 단, 서로 다른 버전의 .NET Framework에 대해 여러 업데이트가 제공될 경우 반드시 따라야 하지는 않지만 낮은 버전에서 높은 버전의 순서대로 업데이트하는 것이 좋습니다.

ASP.NET 패딩 오라클 취약점(CVE-2010-3332)
-----------------------------------------


암호화 패딩 확인 중 부적절한 오류 처리로 인해 ASP.NET에 정보 유출 취약점이 존재합니다. 이 취약점 악용에 성공한 공격자는 View State와 같이 서버에서 암호화된 데이터를 읽을 수 있습니다. 이 취약점 악용에 성공한 경우 서버에서 암호화된 데이터를 해독 및 변조하는 데 사용할 수 있는 데이터 변조에도 이 취약점을 사용할 수 있습니다. 이 취약점으로 인해 공격자가 직접 코드를 실행하거나 해당 사용자 권한을 상승시킬 수는 없지만 영향을 받는 시스템의 손상을 악화시키는 데 사용할 수 있는 정보를 생성할 수 있습니다. Microsoft .NET Framework 3.5 서비스 팩 1 이상에서 공격자는 이 취약점을 악용하여 web.config를 포함한 ASP.NET 응용 프로그램 내의 모든 파일 콘텐츠를 검색할 수도 있습니다.

이 취약점을 CVE(Common Vulnerabilities and Exposures) 목록의 표준 항목으로 보려면 [CVE-2010-3332(영문)](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332)를 참조하십시오.

#### ASP.NET 패딩 오라클 취약점(CVE-2010-3332)에 대한 완화 요소

완화란 취약점의 악용 심각도를 낮출 수 있는 설정, 일반적인 구성 또는 최선책, 기본 상태를 지칭합니다. 다음과 같은 완화 요소가 현재 상황에서 유용할 수 있습니다.

-   Microsoft .NET Framework 3.5 서비스 팩 1 이전의 Microsoft .NET Framework 버전은 이 취약점의 파일 콘텐츠 노출 부분의 영향을 받지 않습니다.

#### ASP.NET 패딩 오라클 취약점(CVE-2010-3332)에 대한 대안

대안이란 기본적인 취약점을 수정하지는 않지만 업데이트를 적용하기 전에 알려진 공격 경로를 차단하는 데 유용한 설정 또는 구성 변경을 지칭합니다. Microsoft는 다음과 같은 대안을 테스트했으며, 대안으로 인한 기능 저하 여부를 해당 항목에서 설명합니다.

-   UrlScan 또는 요청 필터링 규칙 사용, ASP.NET 사용자 지정 오류 사용 및 모든 오류 코드를 동일한 오류 페이지에 매핑

    ASP.NET의 customErrors 기능을 사용하도록 설정하고 항상 동일한 오류 페이지를 반환하도록 응용 프로그램을 명시적으로 구성하면 서버에서 발생하는 오류에 관계 없이 공격자가 현재 악용을 통해 서버에서 발생한 오류의 여러 유형을 구분하는 것이 더 어려워집니다.

    .NET Framework 버전 3.5 서비스 팩 1 이상을 사용하는 시스템의 경우 이 대안을 구현하면 보안이 강화될 뿐만 아니라 현재 악용의 타이밍 공격 부분으로부터 보호할 수도 있습니다. 대안은 customErrors 기능의 redirectMode="ResponseRewrite" 옵션을 사용하며 오류 페이지에 무작위 딜레이를 적용합니다. 이러한 방법을 함께 사용하면 공격자가 오류를 받은 시간을 측정하여 서버에서 발생한 오류의 유형을 추정하기가 어려워집니다.

    또한 이 대안을 구현하려면 쿼리 문자열의 응용 프로그램 오류 경로를 지정하는 요청을 차단해야 합니다. 이 작업은 URLScan이나 관리자가 정의한 규칙에 따라 선별적으로 요청을 차단할 수 있는 IIS(인터넷 정보 서비스)의 무료 도구를 사용하여 수행할 수 있습니다. Windows Vista 서비스 팩 2, Windows Server 2008 서비스 팩 2, Windows 7 또는 Windows Server 2008 R2에서 IIS(인터넷 정보 서비스)를 실행하는 시스템의 경우 요청 필터링 기능을 대신 사용할 수도 있습니다.

    요청 쿼리 문자열의 ASP.Net 응용 프로그램 오류 경로를 수정하는 요청 차단

    UrlScan 사용:

    1. [UrlScan 3.1(영문)](http://www.iis.net/download/urlscan)을 다운로드하고 설치합니다. UrlScan 구성 및 사용 방법에 대한 자세한 내용은 [UrlScan 3 참조(영문)](http://learn.iis.net/page.aspx/938/urlscan-3-reference/)를 참조하십시오.

    2. UrlScan.ini를 수정합니다(%windir%\\system32\\inetsrv\\urlscan에 위치). Urlscan.ini 파일의 \[DenyQueryStringSequences\] 섹션 아래에 다음 줄을 삽입합니다.

    ```
    aspxerrorpath=
    ```

    이 작업을 수행한 후 \[DenyQueryStringSequences\] 섹션이 다음과 비슷하게 표시되어야 합니다(섹션에 추가된 줄은 정상이며 대안에 영향을 주지 않음).

    ```
    [DenyQueryStringSequences]aspxerrorpath=
    ```

    3. 관리자로 로그인한 상태에서 명령 프롬프트에서 iisreset 를 실행합니다.

    IIS 요청 필터링 사용:

    이 지침은 Windows Vista 서비스 팩 2, Windows Server 2008 서비스 팩 2, Windows 7 또는 Windows Server 2008 R2에서 IIS를 실행하는 시스템에서 위의 UrlScan 지침을 대신할 수 있습니다.

    1. 프로그램 추가/제거 또는 역할 관리자에서 IIS(인터넷 정보 서비스), World Wide Web 서비스, 보안의 기능을 선택하여 IIS의 요청 필터링 기능을 설치합니다.

    2. IIS(인터넷 정보 서비스) 관리자를 실행합니다.

    3. 왼쪽 창에서 서버 노드를 선택합니다.

    4. 요청 필터링을 두 번 클릭합니다.

    5. 쿼리 문자열 탭을 클릭하고 작업 창에서 쿼리 문자열 거부…를 클릭합니다.

    6. 대화 상자에 aspxerrorpath=을 입력하고 확인을 선택합니다.

    또는 다음 appcmd 명령을 사용하여 이 요청 쿼리 문자열을 설정할 수도 있습니다.

    ```
    appcmd set config /section:requestfiltering /+denyQueryStringSequences.[sequence='aspxerrorpath=']
    ```

    appcmd를 사용하여 IIS를 구성하는 방법은 [AppCmd.exe로 시작하기(영문)](http://learn.iis.net/page.aspx/114/getting-started-with-appcmdexe)를 참조하십시오.

    동일한 사용자 지정 오류를 사용하도록 ASP.Net 응용 프로그램 구성

    각 ASP.NET 웹 응용 프로그램의 루트 폴더에 이미 web.config 파일이 있는지 확인합니다. 대상 디렉터리에 파일을 만들 수 있는 권한이 있어야 이 대안을 구현할 수 있습니다.

    ASP.NET 응용 프로그램에 web.config 파일이 없는 경우

    .NET Framework 3.5 이전 버전

    1. ASP.NET 응용 프로그램의 루트 폴더에 web.config라는 텍스트 파일을 만들고 다음 내용을 넣습니다.

    ```
    <configuration><location allowOverride="false"><system.web><customErrors mode="On" defaultRedirect="~/error.html" /></system.web></location></configuration>
    ```

    2. 일반 오류 메시지를 포함하는 error.html이라는 텍스트 파일을 만들어 ASP.NET 응용 프로그램의 루트 폴더에 저장합니다.

    3. 또는 기존 오류 페이지를 가리키도록 web.config 파일에 있는 error.html의 이름을 바꿉니다. 단, 해당 페이지에는 컨텍스트별 내용이 아닌 일반 내용이 표시되어야 합니다.

    .NET Framework 3.5 서비스 팩 1 이상 버전

    ```
    <configuration><location allowOverride="false"><system.web><customErrors mode="On" redirectMode="ResponseRewrite" defaultRedirect="~/ErrorPage.aspx" /></system.web></location></configuration>
    ```

이 취약점을 악용하면 어느 작업까지 가능합니까? 
이는 정보 유출 취약점입니다. 이 취약점 악용에 성공한 공격자는 View State와 같이 서버에서 암호화된 데이터를 읽을 수 있습니다. 이 취약점 악용에 성공한 경우 서버에서 암호화된 데이터를 해독 및 변조하는 데 사용할 수 있는 데이터 변조에도 이 취약점을 사용할 수 있습니다. Microsoft .NET Framework 3.5 서비스 팩 1 이상에서 공격자는 이 취약점을 악용하여 web.config를 포함한 ASP.NET 응용 프로그램 내의 모든 파일 콘텐츠를 검색할 수도 있습니다. 이 취약점으로 인해 공격자가 직접 코드를 실행하거나 해당 사용자 권한을 상승시킬 수는 없지만 영향을 받는 시스템의 손상을 악화시키는 데 사용할 수 있는 정보를 생성할 수 있습니다.

취약점의 원인은 무엇입니까? 
ASP.NET의 암호화 패딩을 사용하면 공격자가 암호화된 데이터를 읽고 변조하는 데 사용할 수 있는 오류 응답의 정보를 제공합니다.

ASP.NET이란 무엇입니까? 
ASP.NET은 개발자가 웹 응용 프로그램과 XML 웹 서비스를 구축하는 데 사용하는 Microsoft .NET Framework 내의 기술 집합입니다.

정적 HTML과 스크립팅을 사용하는 기존 웹 페이지와 달리, ASP.NET은 컴파일된 이벤트 구동 페이지를 사용합니다. ASP.NET이 웹 기반 응용 프로그램 환경이므로 기본적인 HTTP 기능성을 제공하기 위해서 기본 웹 서버가 필요하며, ASP.NET은 IIS(인터넷 정보 서비스)에서 실행됩니다. 자세한 내용은 [공식 Microsoft ASP.NET 사이트(영문)](http://www.asp.net/)를 참조하십시오.

ASP.NET View State란 무엇입니까? 
Microsoft ASP.NET View State는 포스트백에 걸쳐 Web Form의 상태에 대한 변경 내용을 유지하기 위해 ASP.NET 웹 페이지에서 사용되는 기법입니다. 페이지의 View State는 기본적으로 \_\_VIEWSTATE라는 숨겨진 양식 필드에 배치되어 있습니다. ASP.NET View State에 대한 자세한 내용은 MSDN 문서, [ASP.NET View State 이해(영문)](http://msdn.microsoft.com/en-us/library/ms972976.aspx)를 참조하십시오.

공격자는 취약점을 악용하여 무엇을 할 수 있습니까? 
이 취약점 악용에 성공한 공격자는 View State와 같이 서버에서 암호화된 데이터를 읽을 수 있습니다. 그러면 공격자는 web.config와 같이 웹 사이트 디렉터리나 하위 디렉터리 내의 모든 파일 콘텐츠를 읽을 수 있습니다. web.config 파일에는 중요한 정보가 저장되는 경우가 많습니다. 이러한 정보 노출의 결과는 정보 자체의 특성에 따라 좌우됩니다.

또한 이 취약점 악용에 공한 공격자는 웹 사이트 디렉터리나 하위 디렉터리에 존재하는 대상 서버의 파일에서 데이터를 읽을 수 있습니다.

공격자는 이러한 취약점을 어떻게 악용합니까? 
이 취약점을 악용하기 위해 공격자는 웹 요청을 통해 영향을 받는 서버에 암호 텍스트를 보내고 웹 사이트에서 반환한 오류 코드를 검사하여 텍스트가 제대로 해독되었는지 여부를 결정합니다. 이러한 요청을 여러 번 시도한 공격자는 충분한 정보를 얻어 암호화된 데이터를 읽거나 변조할 수 있습니다.

취약점으로 인해 주로 공격받을 위험이 있는 시스템은 무엇입니까? 
ASP.NET이 설치된 웹 서버는 이 취약점으로 인한 위험에 노출됩니다.

아래 대안을 사용하는 대신 오류 페이지로 리디렉션하는 사용자 정의 로깅 모듈이 있는 경우에도 취약한 상태가 유지됩니까? 
사용자 정의 로깅 모듈에서 발송되는 응답을 통해 클라이언트가 제공되는 해당 콘텐츠나 시간을 통해 오류 응답을 구분할 수 없는 경우 해당 모듈이 customErrors 대안을 적절히 대체할 수 있습니다. 이러한 응답에는 전체 HTTP 응답과 HTTP 오류 코드가 모두 포함됩니다. 위의 항목이 항상 사실이 아닌 경우 완벽하게 보호되지 않습니다.

이 문제점으로부터 보호하기 위해 모든 기타 오류에 대해 사용자 정의 404 페이지를 만들고 기본 리디렉션으로 설정할 수 있습니까? 
아니요. 그래도 공격자는 404 오류와 기타 오류를 구분할 수 있습니다. 오류를 동질화하는 것은 이 공격을 방지하는 중요한 구성 요소입니다.

View State에 중요한 정보를 저장하지 않은 경우에도 이 취약점을 걱정해야 합니까? 
예. 웹 응용 프로그램의 파일에 중요한 정보가 저장되어 있는 경우 공격자는 이 취약점을 사용하여 해당 파일의 콘텐츠를 유출할 수 있습니다.

web.config 파일 내의 데이터를 보호하는 가장 좋은 방법은 무엇입니까? 
web.config 파일 내의 구성 데이터를 선별적으로 암호화하는 방법이 가장 좋습니다. 구성 섹션을 암호화하는 자세한 내용은 MSDN 문서, [구성 섹션 암호화 및 해독](http://msdn.microsoft.com/ko-kr/library/zhhddkxy(vs.80).aspx)을 참조하십시오.

이 문제가 ASP.NET Web Form 및 ASP.NET MVC 웹 응용 프로그램 모두에 영향을 줍니까? 
예. 이 일반에 공개된 문제는 Web Form 및 MVC를 비롯한 모든 유형의 ASP.NET 응용 프로그램에 대해 악용될 수 있습니다. ASP.NET MVC에 대한 자세한 내용은 MSDN 문서, [ASP.NET MVC 개요(영문)](http://msdn.microsoft.com/en-us/library/dd381412.aspx)를 참조하십시오.

SharePoint 및 Exchange과 같은 ASP.NET 플랫폼에 기반한 기타 응용 프로그램이 이 문제의 영향을 받습니까? 
ASP.NET 플랫폼에 기반한 모든 응용 프로그램은 이 문제의 영향을 받습니다. 이러한 응용 프로그램의 관리자는 이 권고에 기술된 권장 사항을 따르는 것이 좋습니다.

AES 암호화 대신에 Triple DES 암호화로 전환하면 내 ASP.NET 응용 프로그램을 보호하는 데 도움이 됩니까? 
아니요. 제시된 암호화 취약점은 PKCS \#7 패딩을 사용하는 알고리즘의 암호화 패딩 오류를 클라이언트에 노출하는 것과 관련되어 있습니다. Triple DES와 AES는 이 패딩 모드를 공유하므로 Triple DES로 전환해도 이 취약점으로부터 보호할 수 없습니다.

Triple DES(3DES)란 무엇입니까? 
Triple DES(3DES)는 데이터의 각 세그먼트에 3회의 암호화 작업 반복을 사용하는 DES(Data Encryption Standard) 암호화 구현입니다. 각 반복에서는 암호화를 위해 56비트 키를 사용하여 데이터에 대해 168비트 암호화가 만들어집니다. 3DES는 추가적인 암호화 계산으로 인해 DES보다 느리지만 보호 강도는 DES보다 높습니다.

AES(Advanced Encryption Standard)란 무엇입니까? 
AES(Advanced Encryption Standard)는 2001년도에 미국 정부에서 채용한 암호화 방식입니다. AES는 기존 방식인 DES(Data Encryption Standard)보다 더욱 안전한 암호화를 제공합니다.

PKCS \#7 패딩이란 무엇입니까? 
PKCS(공개 키 암호화 표준) \#7은 디지털 서명 및 암호화와 같이 암호화가 적용될 수 있는 데이터의 일반 구문으로, 인증서 또는 인증서 해지 목록 및 기타 메시지 특성(타임 스탬프 등)을 메시지에 전달하기 위한 구문도 제공합니다. PKCS \#7 패딩 문자열은 일련의 바이트로 구성되고 각 바이트는 추가된 총 패딩 바이트 수와 같습니다.

이 문제가 오라클 데이터베이스 제품에 영향을 줍니까? 
아니요. 이 문제는 데이터베이스 제품과 관련이 없습니다. 이 경우에 사용되는 단어인 "오라클"은 암호화 용어를 가리킵니다.

업데이트는 어떤 기능을 수행합니까? 
업데이트는 ASP.NET으로 암호화된 모든 데이터를 서명하여 취약점을 해결합니다.

이 보안 공지를 게시했을 때 이 취약점이 공개되어 있었습니까? 
예. 이 취약점은 공개되었습니다. 이 취약점에는 CVE(Common Vulnerability and Exposure) 번호 [CVE-2010-3332(영문)](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332)가 지정되었습니다.

이 보안 공지를 게시했을 때 Microsoft는 이 취약점이 악용되었다는 보고를 받았습니까? 
예. Microsoft는 이 취약점을 악용하려는 제한적이며 대상이 일정한 공격에 대한 보고를 받았습니다.

UCE(Update Compatibility Evaluator) 및 ACT(Application Compatibility Toolkit)

업데이트는 응용 프로그램을 실행하는 데 필요한 같은 파일 및 레지스트리 설정에 기록하는 경우가 많습니다. 이에 따라 비호환성이 발생하고 보안 업데이트 배포 시간이 길어질 수 있습니다. [ACT(Application Compatibility Toolkit)(영문)](http://www.microsoft.com/downloads/details.aspx?familyid=24da89e9-b581-47b0-b45e-492dd6da2971&displaylang=en)에 포함된 [UCE(Update Compatibility Evaluator)(영문)](http://technet2.microsoft.com/windowsvista/en/library/4279e239-37a4-44aa-aec5-4e70fe39f9de1033.mspx?mfr=true) 구성 요소를 사용하면 설치된 응용 프로그램에 대한 Windows 업데이트 테스트 및 유효성 검사를 능률화할 수 있습니다.

ACT(Application Compatibility Toolkit)에는 Microsoft Windows Vista, Windows Update, Microsoft 보안 업데이트 또는 Windows Internet Explorer의 새 버전을 사용자 환경에 배포하기 전에 응용 프로그램 호환성 문제점을 평가 및 완화하는 데 필요한 도구와 설명서가 들어 있습니다.

적용된 업데이트 확인

-   Microsoft Baseline Security Analyzer

    MBSA(Microsoft Baseline Security Analyzer) 도구를 사용하면 영향을 받는 시스템에 보안 업데이트가 적용되었는지 확인할 수 있습니다. 자세한 내용은 이 공지 앞부분에 있는 검색, 탐지 도구 및 지침 섹션을 참조하십시오.

-   파일 버전 확인

    Microsoft Windows에는 여러 버전이 있으므로 사용자의 컴퓨터에서 다음 단계가 다르게 적용될 수 있습니다. 이러한 경우 제품 설명서를 참조하여 작업 단계를 완료하십시오.

    1.  시작, 검색을 차례로 클릭합니다.
    2.  검색 결과 창의 검색 도우미에서 모든 파일 및 폴더를 클릭합니다.
    3.  전체 또는 일부 파일 이름 상자 에 적절한 파일 정보 표의 파일 이름을 입력한 다음 검색을 클릭합니다.
    4.  파일 목록에서 해당 파일 정보 표의 파일 이름을 마우스 오른쪽 단추로 클릭한 다음 속성을 클릭합니다.
        참고 운영 체제 에디션 또는 시스템에 설치된 프로그램에 따라 파일 정보 표에 나열된 파일 중 일부가 설치되어 있지 않을 수도 있습니다.
    5.  버전 탭의 버전을 해당 파일 정보 표에 있는 버전과 비교하여 사용자 시스템에 설치된 파일 버전을 확인합니다.
    6.  참고 파일 버전 이외의 특성은 설치 중 변경될 수 있습니다. 다른 파일 특성과 파일 정보 표에 있는 정보를 비교하는 것은 지원되는 업데이트 적용 확인 방법이 아닙니다. 경우에 따라 파일 이름이 설치 도중 변경될 수도 있습니다. 파일 또는 버전 정보가 없으면 사용 가능한 다른 방법 중 하나를 사용하여 업데이트 설치를 확인하십시오.

-   레지스트리 키 확인

    이 섹션의 참조 표에 나열된 레지스트리 키를 검토하여 이 보안 업데이트가 설치한 파일을 확인할 수도 있습니다.

    이 레지스트리 키에 설치된 파일의 전체 목록 중 일부가 들어 있지 않을 수 있습니다. 또한 관리자나 OEM이 보안 업데이트를 Windows 설치 원본 파일에 통합하거나 포함시키는 경우 이러한 레지스트리 키가 제대로 만들어지지 않을 수 있습니다.

적용된 업데이트 확인

-   Microsoft Baseline Security Analyzer

    MBSA(Microsoft Baseline Security Analyzer) 도구를 사용하면 영향을 받는 시스템에 보안 업데이트가 적용되었는지 확인할 수 있습니다. 자세한 내용은 이 공지 앞부분에 있는 검색, 탐지 도구 및 지침 섹션을 참조하십시오.

-   파일 버전 확인

    Microsoft Windows에는 여러 버전이 있으므로 사용자의 컴퓨터에서 다음 단계가 다르게 적용될 수 있습니다. 이러한 경우 제품 설명서를 참조하여 작업 단계를 완료하십시오.

    1.  시작, 검색을 차례로 클릭합니다.
    2.  검색 결과 창의 검색 도우미에서 모든 파일 및 폴더를 클릭합니다.
    3.  전체 또는 일부 파일 이름 상자 에 적절한 파일 정보 표의 파일 이름을 입력한 다음 검색을 클릭합니다.
    4.  파일 목록에서 해당 파일 정보 표의 파일 이름을 마우스 오른쪽 단추로 클릭한 다음 속성을 클릭합니다.
        참고 운영 체제 에디션 또는 시스템에 설치된 프로그램에 따라 파일 정보 표에 나열된 파일 중 일부가 설치되어 있지 않을 수도 있습니다.
    5.  버전 탭의 버전을 해당 파일 정보 표에 있는 버전과 비교하여 사용자 시스템에 설치된 파일 버전을 확인합니다.
    6.  참고 파일 버전 이외의 특성은 설치 중 변경될 수 있습니다. 다른 파일 특성과 파일 정보 표에 있는 정보를 비교하는 것은 지원되는 업데이트 적용 확인 방법이 아닙니다. 경우에 따라 파일 이름이 설치 도중 변경될 수도 있습니다. 파일 또는 버전 정보가 없으면 사용 가능한 다른 방법 중 하나를 사용하여 업데이트 설치를 확인하십시오.

-   레지스트리 키 확인

    이 섹션의 참조 표에 나열된 레지스트리 키를 검토하여 이 보안 업데이트가 설치한 파일을 확인할 수도 있습니다.

    이 레지스트리 키에 설치된 파일의 전체 목록 중 일부가 들어 있지 않을 수 있습니다. 또한 관리자나 OEM이 보안 업데이트를 Windows 설치 원본 파일에 통합하거나 포함시키는 경우 이러한 레지스트리 키가 제대로 만들어지지 않을 수 있습니다.

이 보안 업데이트를 설치할 때 설치 관리자는 해당 컴퓨터에서 하나 이상의 업데이트할 파일이 Microsoft 핫픽스에 의해 이미 업데이트되었는지 확인합니다.

핫픽스와 같이 이 게시판에 나타나는 용어에 대한 자세한 내용은 [Microsoft 기술 자료 문서 824684](http://support.microsoft.com/kb/824684)를 참조하십시오.

이 보안 업데이트는 다음 설치 스위치를 지원합니다.

| 스위치        | 설명                                                                                                           |
|---------------|----------------------------------------------------------------------------------------------------------------|
| /?, /h, /help | 지원되는 스위치에서 도움말을 표시합니다.                                                                       |
| /quiet        | 상태 표시 또는 오류 메시지를 표시하지 않습니다.                                                                |
| /norestart    | /quiet와 함께 사용하면 설치 완료를 위해 다시 시작이 필수적인 경우에도 시스템이 설치 후 다시 시작되지 않습니다. |

참고 wusa.exe 설치 프로그램에 대한 자세한 내용은 [Microsoft 기술 자료 문서 934307](http://support.microsoft.com/kb/934307)을 참조하십시오.

적용된 업데이트 확인

-   Microsoft Baseline Security Analyzer

    MBSA(Microsoft Baseline Security Analyzer) 도구를 사용하면 영향을 받는 시스템에 보안 업데이트가 적용되었는지 확인할 수 있습니다. 자세한 내용은 이 공지 앞부분에 있는 검색, 탐지 도구 및 지침 섹션을 참조하십시오.

-   파일 버전 확인

    Microsoft Windows에는 여러 버전이 있으므로 사용자의 컴퓨터에서 다음 단계가 다르게 적용될 수 있습니다. 이러한 경우 제품 설명서를 참조하여 작업 단계를 완료하십시오.

    1.  시작을 클릭한 다음 검색 시작에 업데이트 파일 이름을 입력합니다.
    2.  프로그램에 해당 파일이 나타나면 파일 이름을 마우스 오른쪽 단추로 클릭하고 속성을 클릭합니다.
    3.  일반 탭에 표시된 파일 크기를 공지 KB 문서에 제공된 파일 정보 표와 비교합니다.
    4.  세부 정보 탭 및 비교 정보를 클릭하여 파일 버전과 수정된 날짜를 공지 KB 문서에 제공된 파일 정보 표와 비교할 수도 있습니다.
    5.  마지막으로, 이전 버전 탭을 클릭하여 표시된 이전 파일 버전의 파일 정보를 새 파일 또는 업데이트된 파일 버전의 파일 정보와 비교할 수도 있습니다.

#### 배포 정보
  
업데이트 설치
  
이 보안 업데이트를 설치할 때 설치 관리자는 해당 컴퓨터에서 하나 이상의 업데이트할 파일이 Microsoft 핫픽스에 의해 이미 업데이트되었는지 확인합니다.
  
핫픽스와 같이 이 게시판에 나타나는 용어에 대한 자세한 내용은 [Microsoft 기술 자료 문서 824684](http://support.microsoft.com/kb/824684)를 참조하십시오.
  
이 보안 업데이트는 다음 설치 스위치를 지원합니다.

지원되는 보안 업데이트 설치 스위치

| 스위치        | 설명                                                                                                           |  
|---------------|----------------------------------------------------------------------------------------------------------------|  
| /?, /h, /help | 지원되는 스위치에서 도움말을 표시합니다.                                                                       |  
| /quiet        | 상태 표시 또는 오류 메시지를 표시하지 않습니다.                                                                |  
| /norestart    | /quiet와 함께 사용하면 설치 완료를 위해 다시 시작이 필수적인 경우에도 시스템이 설치 후 다시 시작되지 않습니다. |
  
참고 wusa.exe 설치 프로그램에 대한 자세한 내용은 [Microsoft 기술 자료 문서 934307](http://support.microsoft.com/kb/934307)을 참조하십시오.
  
적용된 업데이트 확인
  
-   Microsoft Baseline Security Analyzer
  
    MBSA(Microsoft Baseline Security Analyzer) 도구를 사용하면 영향을 받는 시스템에 보안 업데이트가 적용되었는지 확인할 수 있습니다. 자세한 내용은 이 공지 앞부분에 있는 검색, 탐지 도구 및 지침 섹션을 참조하십시오.
  
-   파일 버전 확인
  
    Microsoft Windows에는 여러 버전이 있으므로 사용자의 컴퓨터에서 다음 단계가 다르게 적용될 수 있습니다. 이러한 경우 제품 설명서를 참조하여 작업 단계를 완료하십시오.
  
    1.  시작을 클릭한 다음 검색 시작에 업데이트 파일 이름을 입력합니다.  
    2.  프로그램에 해당 파일이 나타나면 파일 이름을 마우스 오른쪽 단추로 클릭하고 속성을 클릭합니다.  
    3.  일반 탭에 표시된 파일 크기를 공지 KB 문서에 제공된 파일 정보 표와 비교합니다.  
    4.  세부 정보 탭 및 비교 정보를 클릭하여 파일 버전과 수정된 날짜를 공지 KB 문서에 제공된 파일 정보 표와 비교할 수도 있습니다.  
    5.  마지막으로, 이전 버전 탭을 클릭하여 표시된 이전 파일 버전의 파일 정보를 새 파일 또는 업데이트된 파일 버전의 파일 정보와 비교할 수도 있습니다.
  
### 기타 정보
  
#### MAPP(Microsoft Active Protections Program)
  
고객에 대한 보안을 강화하기 위해 Microsoft는 월별 보안 업데이트를 배포하기 전에 주요 보안 소프트웨어 제공업체에 취약점 정보를 제공합니다. 보안 소프트웨어 제공업체는 이 취약점 정보를 사용하여 안티바이러스, 네트워크 기반 침입 탐지 시스템 또는 호스트 기반 침임 방지 시스템 등 자사의 보안 소프트웨어나 장치를 통해 업데이트된 보호 기능을 고객에게 제공할 수 있습니다. 보안 소프트웨어 제공업체가 활성 보호 기능을 제공하는지 확인하려면 [Microsoft MAPP(Active Protections Program) 파트너](http://www.microsoft.com/korea/security/msrc/mapp/partners.mspx)에 나열된 프로그램 파트너가 제공하는 활성 보호 기능 웹 사이트를 참조하십시오.

Microsoft Corporation 또는 그 공급자는 모든 직접적, 간접적, 부수적, 파생적 손해 또는 영업 이익 손실 또는 특수한 손실에 대하여 어떠한 경우에도 책임을 지지 않으며, 이는 Microsoft Corporation과 그 공급자가 그와 같은 손해의 가능성을 사전에 알고 있던 경우에도 마찬가지입니다. 일부 지역에서는 파생적 또는 부수적 손해에 대한 책임의 배제 또는 제한을 허용하지 않으므로 위 제한은 귀하에게 적용되지 않을 수도 있습니다.
