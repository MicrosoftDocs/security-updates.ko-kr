Microsoft 보안 공지 MS04-044 - 중요
===================================

Windows 커널 및 LSASS 취약점으로 인한 권한 상승 문제(885835)
------------------------------------------------------------

게시된 날짜: 2004년 12월 14일 화요일

**취약점으로 인한 영향:** 권한 상승

**최대 위험 등급:** 중요

취약점 세부 정보
----------------

#### Windows 커널 취약점 - CAN-2004-0893:

Windows 커널에 응용 프로그램을 실행하는 방식에 [권한 상승](https://www.microsoft.com/korea/security/glossary/glossary_a_z.asp) 취약점이 존재합니다. 이 취약점을 통해 로그온한 사용자가 시스템에 대한 전체 제어 권한을 얻을 수 있습니다.

#### Windows 커널 취약점 - CAN-2004-0893에 대한 완화 요소:

-   이 취약점을 악용하기 위해서는 침입자가 유효한 로그온 자격 증명을 가지고 로컬로 로그온할 수 있어야 합니다. 익명의 사용자나 원격에서는 이 취약점을 악용할 수 없습니다.
-   Windows XP 서비스 팩 2 또는 Windows Server 2003이 실행 중인 시스템에서 이 취약점을 악용하면 서비스 거부가 발생할 가능성이 큽니다.

#### Windows 커널 취약점 - CAN-2004-0893에 대한 대안:

이 취약점에 대한 대안을 아직 알아내지 못했습니다.

#### Windows 커널 취약점 - CAN-2004-0893에 대한 FAQ:

**이 취약점을 악용하면 어느 작업까지 가능합니까?**  
이것은 [권한 상승](https://www.microsoft.com/korea/security/glossary/glossary_a_z.asp) 취약점입니다. 이 취약점을 성공적으로 악용하는 경우 침입자는 프로그램 설치, 데이터 보기, 변경 또는 삭제, 모든 권한을 가진 새 계정 만들기 등의 작업을 포함하여 해당 시스템을 완전히 제어할 수 있습니다. 취약점을 악용하기 위해서는 침입자가 로컬로 시스템에 로그온하고 프로그램을 실행할 수 있어야 합니다.

**취약점의 원인은 무엇입니까?**  
LPC(로컬 프로시저 호출) 인터페이스의 검사되지 않은 버퍼가 LPC 포트를 통해 전송된 데이터 크기에 대한 제한 사항의 유효성을 검사하지 않습니다.

**LPC란 무엇입니까?**  
LPC는 Windows를 통해 제공되는 메시지 전달 서비스입니다. 이 서비스를 통해 스레드와 프로세스가 상호 통신할 수 있습니다. 클라이언트 프로세스가 서버 프로세스로부터 서비스를 요청하는 경우 두 가지 프로세스가 상호 통신할 수 있는 방법이 있어야 합니다. 클라이언트 프로세스가 서버에 요청을 하고, 서버가 클라이언트에 응답을 전송하며, 서로 상태를 확인할 수 있는 방법이 있어야 합니다. 클라이언트 프로세스와 서버 프로세스가 다른 시스템에 있는 경우 RPC(원격 프로시저 호출)가 사용됩니다. 같은 시스템에 있을 때는 LPC를 사용합니다.

**LPC 포트란?**  
모든 LPC에는 LPC 포트라고 하는 통신 채널 모음이 있습니다. 각 포트는 한 가지 종류의 통신을 수행합니다 예를 들어, LPC에는 항상 클라이언트가 서버에 메시지를 보내는 데 사용하는 포트와 서버가 각 클라이언트에 메시지를 보내는 포트가 있습니다. 또한, LPC는 프로세스 내의 스레드가 이러한 요청을 조정하는 각기 다른 목적을 위한 포트를 가지기도 합니다.

**침입자는 취약점을 악용하여 무엇을 할 수 있습니까?**  
이 취약점을 성공적으로 악용하는 경우 침입자는 프로그램 설치, 데이터 보기, 변경 또는 삭제, 모든 권한을 가진 새 계정 만들기 등의 작업을 포함하여 해당 시스템을 완전히 제어할 수 있습니다.

**누가 취약점을 악용할 수 있습니까?**  
취약점을 악용하기 위해서는 침입자가 로컬로 시스템에 로그온하고 프로그램을 실행할 수 있어야 합니다.

**Systems Management Server**

Microsoft Systems Management Server(SMS)는 업데이트 관리를 위해 효율적으로 구성할 수 있는 엔터프라이즈 솔루션을 제공합니다. SMS를 사용하면 관리자는 보안 업데이트가 필요한 Windows 기반 시스템을 식별하고, 사용자의 업무 중단을 최소화하면서 엔터프라이즈 전체에 해당 업데이트를 효율적으로 배포할 수 있습니다. 관리자가 SMS 2003을 사용하여 보안 업데이트를 배포하는 방법에 대한 자세한 내용은 [SMS 2003 보안 패치 관리 웹 사이트](https://www.microsoft.com/korea/smserver/evaluation/capabilities/patch.asp)를 참조하십시오. SMS 2.0 사용자는 [Software Updates Service 기능 팩 (US)](https://www.microsoft.com/smserver/downloads/20/featurepacks/suspack/)을 사용하여 보안 업데이트를 배포할 수도 있습니다. SMS에 대한 내용은 [SMS 웹 사이트](https://www.microsoft.com/korea/smserver/default.asp)를 참조하십시오.

**참고** SMS에서는 Microsoft Baseline Security Analyzer와 Microsoft Office Detection Tool을 사용하여 보안 게시판 업데이트 검색 및 배포와 관련된 다양한 기능을 지원합니다. 이러한 도구로 일부 소프트웨어 업데이트를 검색하지 못할 수도 있습니다. 관리자는 특정 시스템을 업데이트하려는 경우 SMS의 인벤토리 기능을 사용할 수 있습니다. 이 절차에 대한 자세한 내용은 [여기 (US)](https://www.microsoft.com/technet/prodtechnol/sms/sms2003/patchupdate.mspx)를 참조하십시오. 일부 보안 업데이트 작업에는 시스템을 다시 시작한 후에 관리자 권한이 필요합니다. 관리자는 Elevated Rights Deployment Tool을 사용하여 이러한 업데이트를 설치할 수 있으며, 이 도구는 [**SMS 2003 Administration 기능 팩 (US)**](https://www.microsoft.com/smserver/downloads/2003/adminpack.asp) 및 [**SMS 2.0 Administration 기능 팩 (US)**](https://www.microsoft.com/smserver/downloads/20/featurepacks/adminpack/)에서 구할 수 있습니다.
